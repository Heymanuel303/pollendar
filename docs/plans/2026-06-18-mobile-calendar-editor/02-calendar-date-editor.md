# Phase 2: Calendar date editor

**Plan:** [2026-06-18-mobile-calendar-editor](00-overview.md)
**Depends on:** 01-slot-presets.md
**Execution:** solo

## Context
We're replacing the one-date-at-a-time candidate-times editor with a calendar multi-select + bulk-slot-apply flow (mobile + desktop) behind a future `Calendar | List` toggle, while emitting the identical `CreatePollPayload` (no backend change). This phase builds the calendar half of that flow: a tap-to-multi-select month grid that maps selected dates into `dates[]` of `PollDateInput`. The component is fully controlled (`v-model`) over the same `PollDateInput[]` array `DateSlotEditor` already drives, so Phase 3 can swap between the two views over one shared `dates` ref in `PollEditor.vue` with no payload divergence.

## Objective
Build `CalendarDateEditor.vue`: a month grid with tap-to-multi-select dates (not a range picker), month navigation, a live "N selected" count, and a bulk-apply panel (using `SlotPresetChips`) that maps selected dates into `dates[]` of `PollDateInput`.

## Steps
1. **Scaffold the controlled shell.** Create `frontend/src/components/CalendarDateEditor.vue` with `<script setup lang="ts">`. Import `{ computed, ref } from 'vue'`, `SlotPresetChips from '@/components/SlotPresetChips.vue'`, the preset source from `@/lib/slotPresets`, `{ formatDate, formatDayNumber, isoDateParts, nextCandidateDate } from '@/lib/utils/timezone'`, and `type { PollDateInput, PollSlotInput } from '@/types/poll'`. Declare `defineProps<{ modelValue: PollDateInput[]; timezone: string; showErrors?: boolean }>()` and `const emit = defineEmits<{ 'update:modelValue': [PollDateInput[]] }>()` — byte-for-byte the same contract as `DateSlotEditor.vue` lines 13–19, so Phase 3 can mount either over the same `v-model="dates"`.

2. **Derive the current selection from `modelValue` (single source of truth).** Add `const selectedSet = computed<Set<string>>(() => new Set(props.modelValue.map((d) => d.eventDate.slice(0, 10))))`. Use `.slice(0, 10)` (consistent with `isoDateParts`) so a date is treated as selected whether stored as bare `'YYYY-MM-DD'` or a full ISO instant. The grid reads selection from this computed — never a parallel local `ref` — so the component stays a pure function of its props (same controlled discipline as `DateCard.vue` / `SlotRow.vue`). Add `const selectedCount = computed(() => props.modelValue.length)` for the live badge.

3. **Month-navigation state (the only local UI ref).** Seed the visible month from the first selected date, else today: parse the first `modelValue[0]?.eventDate` via `isoDateParts` (or `new Date()` when empty) and store `const cursorYear = ref<number>(…)`, `const cursorMonth = ref<number>(…)` (0-indexed). Add `prevMonth()` / `nextMonth()` that adjust these refs with month-rollover (Dec→Jan increments year, Jan→Dec decrements), e.g. via `new Date(Date.UTC(cursorYear.value, cursorMonth.value ± 1, 1))` then read back `getUTCFullYear()` / `getUTCMonth()`. This is presentation-only state — it does NOT emit. A `monthLabel` computed formats the header as `"June 2026"` via `new Intl.DateTimeFormat('en-US', { month: 'long', year: 'numeric', timeZone: 'UTC' }).format(new Date(Date.UTC(cursorYear.value, cursorMonth.value, 1)))`.

4. **Build the grid cells (timezone-safe, no day-shift).** Add `const cells = computed(() => …)` returning the calendar matrix for the cursor month: compute the weekday of the 1st via `new Date(Date.UTC(cursorYear.value, cursorMonth.value, 1)).getUTCDay()`, the day count via `new Date(Date.UTC(cursorYear.value, cursorMonth.value + 1, 0)).getUTCDate()`, and emit an array of `{ iso: string | null }` (leading `null` blanks for grid alignment, then each day as a bare `iso = `${year}-${MM}-${DD}`` zero-padded). Build the `iso` string from the cursor numbers directly — do NOT round-trip through `Date.toISOString()` for the cell key, to avoid any UTC-offset day-shift. Each non-blank cell carries `selected: selectedSet.value.has(cell.iso)`. Render a 7-column grid with `Su…Sa` weekday headers; render day cells as `<button type="button" class="touch-target …">` so each tap zone is ≥44×44px (the `.touch-target` utility from responsive-foundations); use `.num` for the day number and `bg-pollen text-canvas` (or `.pollen-dot` marker) for the selected state vs `bg-canvas text-dim` unselected. Disable past dates if desired by comparing `iso < nextCandidateDate()`-style boundary (optional; keep simple — at minimum do not block today).

5. **Tap-to-toggle a single date (emit immutable replacement).** Add `function toggleDate(iso: string): void`. If `selectedSet.value.has(iso)`, emit `modelValue.filter((d) => d.eventDate.slice(0, 10) !== iso)` (deselect → drop that date). Otherwise emit `[...modelValue, { eventDate: iso, slots: defaultSlots() }]` where `defaultSlots()` returns the current bulk-apply preset's slots (see step 6) — or, if no preset chosen yet, the same single seed slot the existing editors use: `[{ startTime: '18:00', endTime: '20:00', isAllDay: false }]` (matches `DateSlotEditor.addDate` line 46 / `DateCard.addSlot` line 39). Keep newly-added dates sorted ascending by `eventDate` before emitting (e.g. `.sort((a, b) => a.eventDate.localeCompare(b.eventDate))`) so the list reads chronologically; this is display ordering only and **must not** set `sortOrder`. Each toggle re-emits the full new array; the parent re-feeds via `v-model` and the grid recomputes `selectedSet` — same controlled round-trip `DateSlotEditor` uses (`updateDate` → parent → recompute).

6. **Bulk-apply panel (the `SlotPresetChips` integration).** Below the grid, render a panel that mounts `<SlotPresetChips … />` (consume its emit per Phase 1's contract — a chosen preset / `PollSlotInput[]`). Keep the active preset in a local `const activeSlots = ref<PollSlotInput[]>(…)`, defaulting to the existing seed slot. `defaultSlots()` (step 5) returns a fresh **copy** of `activeSlots.value` (`activeSlots.value.map((s) => ({ ...s }))`) so each date gets its own slot objects (no shared reference). Add an "Apply to N selected" button: on click, emit `props.modelValue.map((d) => ({ eventDate: d.eventDate, slots: activeSlots.value.map((s) => ({ ...s })) }))` — rewriting **every currently selected date's** slots to the active preset, immutably, preserving each `eventDate`. This is the multi-select bulk-apply: select many days, pick a preset, apply once. The button is disabled when `selectedCount === 0`. Disable/guard so applying never introduces `sortOrder` or `closesAt`.

7. **Live count + empty/error affordances.** Render the "N selected" badge using `selectedCount` (reuse `Pill.vue` `tone="pollen"` or mirror its pattern — `<span class="num">{{ selectedCount }}</span> selected`). When `props.showErrors === true && selectedCount === 0`, render the same coral "Add at least one candidate date." inline message `DateSlotEditor.vue` shows (lines 77–79, `text-coral`), so error parity carries across both views.

8. **Slot shape parity — the payload guarantee.** Ensure every emitted `PollDateInput` is exactly `{ eventDate: '<YYYY-MM-DD>', slots: PollSlotInput[] }` — bare 10-char date string, no `sortOrder` on the date, no `sortOrder` on any slot, no `closesAt` anywhere. Time-range slots emit `{ startTime, endTime, isAllDay: false }`; all-day presets emit `{ isAllDay: true }` (no times). This is the identical element shape `DateSlotEditor`/`DateCard`/`SlotRow` produce, so when Phase 3 feeds this component's emitted array into the unchanged `buildPayload()` (`PollEditor.vue` lines 73–88), the resulting `CreatePollPayload.dates[]` is byte-identical to the List flow's. **Do not** call `pollStore.create()` or touch `buildPayload()` here — this component only owns the `dates[]` array.

9. **Use `isPhone` for layout density only.** Import `useBreakpoint` from `@/composables/useBreakpoint` and read `isPhone` to tune density (e.g. larger cells / stacked bulk-apply panel on phone, inline on desktop) — purely cosmetic. The selection logic, emitted shape, and `dates[]` contract are identical across breakpoints; never branch the payload on viewport. If the composable is not yet on disk (responsive-foundations Phase 1), the import path is still correct — keep the usage minimal so a missing-on-disk dependency surfaces as a clear build error rather than divergent behavior.

10. **Write `CalendarDateEditor.spec.ts`.** Copy the harness shape from `DateSlotEditor.spec.ts` (lines 1–41): `mount()` with `props: { modelValue, timezone: 'Europe/Brussels' }`, `lastModel(wrapper)` to read the newest `update:modelValue`, `buttonByText(wrapper, …)` to find action buttons. Assert: (a) tapping an unselected day emits a `modelValue` grown by one date whose `eventDate` is a bare `'YYYY-MM-DD'` and whose `slots[0].isAllDay === false`; (b) tapping an already-selected day emits the array with that `eventDate` removed; (c) the rendered count text reflects `modelValue.length` after `setProps`; (d) "Apply to N selected" rewrites every selected date's `slots` to the active preset while leaving `eventDate`s unchanged; (e) no emitted date or slot carries a `sortOrder` key and no `closesAt` appears. Stub/mount `SlotPresetChips` per Phase 1's exported test contract if needed.
