# Phase 3: Poll editor — create a poll with candidate dates + slots

**Plan:** [frontend-build-out](00-overview.md)
**Depends on:** 02-auth-flow.md
**Execution:** solo

## Context
Pollendar's frontend is a Vue 3.5 + Vite + TypeScript SPA that lets a signed-in creator gather everyone's availability and pick the time that works. This phase delivers the **PollEditor** view at `/polls/new` — the screen a creator uses to draft a poll (title, description, timezone, optional close date) plus a nested set of candidate dates, each with one or more time-range or all-day slots. It is the first authenticated, data-mutating creator flow; Phase 2 already established the auth cookie session, the `apiFetch` client (`credentials: "include"`), the router guard that redirects unauthenticated users to `/dashboard`, and the `authStore`. Everything is styled to the "Dusk Calendar" dark theme and built under the **Tailwind v4 CSS-first** constraint (tokens in the `@theme` block of `src/assets/main.css`; motif helpers `.bg-dusk` / `.pollen-dot` / `.bloom` / `.bloom-bg` as custom classes — **no `tailwind.config.js`, no PostCSS, no v3 `@tailwind` directives**). All numerals (dates, times, counts) render in the `--font-display` family via a `.num` utility.

## Objective
Build the `PollEditor` view + `DateSlotEditor` (and a small `DateCard`/slot-row sub-structure) + `pollStore.create()`, so a signed-in creator can compose and POST a valid poll to `POST /api/polls`, then land on the new poll's manage screen with its share URL captured.

## Files to touch
- `frontend/src/types/poll.ts` — shared TS types (NEW or extend): `PollSlotInput` (`{ startTime?: string; endTime?: string; isAllDay?: boolean; label?: string; sortOrder?: number }`), `PollDateInput` (`{ eventDate: string; sortOrder?: number; slots: PollSlotInput[] }`), `CreatePollPayload` (`{ title: string; description?: string; timezone?: string; dates: PollDateInput[] }`), and `CreatedPoll` (`{ id: string; publicToken: string; shareUrl: string; title: string; status: 'open' }`). **All ids typed `string`** (BigInt serialized as strings by the backend interceptor).
- `frontend/src/lib/datetime.ts` — shared tz/format util (NEW or extend): `defaultTimezone()` via `Intl.DateTimeFormat().resolvedOptions().timeZone`; `commonTimezones()` returning `Intl.supportedValuesOf('timeZone')` with a hardcoded fallback list; `formatDateLabel(isoDate, tz)` → "Thu Jun 26"; `formatDayNumber(isoDate)` → "26"; `formatCloseLabel(iso, tz)` → "Thu Jun 25, 18:00". Uses native `Intl.DateTimeFormat` only — no extra tz dependency.
- `frontend/src/stores/pollStore.ts` — Pinia store (NEW): add `create(payload: CreatePollPayload): Promise<CreatedPoll>` (POST `/api/polls`), exposing `creating`/`error` state. (Other CRUD — `findOne`, `update`, `complete`, `remove` — lands in later phases; stub only what this phase needs.)
- `frontend/src/components/DateSlotEditor.vue` — NEW: renders the candidate-dates list (collapsible/`DateCard` rows), add-date / add-slot / remove buttons, all-day vs time-range toggle, slot label input, and the reactive "N dates · M slots" badge; `v-model`-style two-way binding of a `PollDateInput[]` array.
- `frontend/src/components/DateCard.vue` — NEW (split out of DateSlotEditor for maintainability): one candidate date's card — day-number chip, formatted date label, remove-date button, slot rows, add-slot button.
- `frontend/src/components/SlotRow.vue` — NEW (split out): one slot — label input, all-day/set-times toggle, start/end time inputs (`type="time"`, HH:mm), per-slot validation error text, remove-slot button.
- `frontend/src/components/ui/FormField.vue` — NEW primitive (if not already created in Phase 2): label + slot + optional hint/error text, matching the form-fields mockup (`label` styling, `text-coral` error). Reuse if it exists.
- `frontend/src/components/ui/TimezoneSelect.vue` — NEW: styled `<select>` over `commonTimezones()`, `v-model` string.
- `frontend/src/views/PollEditor.vue` — NEW: thin view composing the form section (title/description/timezone/closesAt via FormField + TimezoneSelect), `DateSlotEditor`, the sticky **Preview** aside, and the primary "Create poll" action; delegates POST to `pollStore.create()` and navigates on success.
- `frontend/src/router/index.ts` — add the `/polls/new` route (name `poll-new`, `meta: { requiresAuth: true }`, lazy `() => import('../views/PollEditor.vue')`). (The auth guard itself is owned by Phase 2 — only register the route here.)
- `frontend/src/components/__tests__/DateSlotEditor.spec.ts` — NEW unit test (Vitest + @vue/test-utils) for add/remove and all-day toggle behavior.

## Steps
1. **Types** — in `frontend/src/types/poll.ts`, declare `PollSlotInput`, `PollDateInput`, `CreatePollPayload`, and `CreatedPoll` exactly mirroring `CreatePollDto`/`CreatePollSlotDto`/`CreatePollDateDto` (`backend/src/polls/dto/create-poll.dto.ts`) and the controller's thin 201 shape (`backend/src/polls/polls.controller.ts` `create()` → `{ id, publicToken, shareUrl, title, status }`). Keep `closesAt` OUT of `CreatePollPayload` — it is **PATCH-only** (`update-poll.dto.ts`), not accepted on create.
2. **Datetime util** — in `frontend/src/lib/datetime.ts`, implement `defaultTimezone()`, `commonTimezones()` (try `Intl.supportedValuesOf('timeZone')`, catch → hardcoded list incl. `UTC`, `Europe/Brussels`, `Europe/London`, `America/New_York`, `Asia/Tokyo`), `formatDateLabel`, `formatDayNumber`, `formatCloseLabel` — all via `new Intl.DateTimeFormat(undefined, { timeZone, … })`. No new dependency.
3. **TimezoneSelect** — in `frontend/src/components/ui/TimezoneSelect.vue`, render the appearance-none `<select>` from the form-fields mockup (`rounded-xl border-line bg-canvas px-4 py-3 pr-10 … focus:border-pollen focus:ring-2 focus:ring-pollen/30`) populated from `commonTimezones()`, `v-model` a string. Default selection comes from the parent (PollEditor sets it to `defaultTimezone()`).
4. **SlotRow** — in `frontend/src/components/SlotRow.vue`, render one slot row matching `docs/design/mockups/components/date-slot-editor.html`: label input (`aria-label="Slot label"`), an All-day / Set-times segmented toggle, and `<input type="time">` start/end (HH:mm, `.num`/`font-display`, separated by an en-dash). When `isAllDay` is true, hide start/end and show the "No fixed time" label (per `poll-editor.html`). Emit slot changes upward; expose a `remove` event and a validation flag. **Do not** render the static tally pollen-dots from the component mockup — those illustrate post-vote results, not create-mode input.
5. **DateCard** — in `frontend/src/components/DateCard.vue`, render one candidate-date card from `poll-editor.html`: a `bg-surface2` chip showing `formatDayNumber(eventDate)` in `font-display text-pollen`, the `formatDateLabel(eventDate, timezone)` heading, a live "N slot(s)" subcount, a remove-date button (`hover:text-coral`), the list of `SlotRow`s, and a "+ Add slot" button that appends a default time-range slot (e.g. `{ startTime: '18:00', endTime: '20:00', isAllDay: false }`). Pass `timezone` down so the date label reflects the chosen zone.
6. **DateSlotEditor** — in `frontend/src/components/DateSlotEditor.vue`, accept `modelValue: PollDateInput[]` + `timezone: string`; render the "Candidate times" section header, the reactive `"<n> dates · <m> slots"` badge (`m` = total slots across all dates), a list of `DateCard`s, and the dashed "+ Add date" button that appends a new `PollDateInput` (one default slot, `eventDate` defaulted to a sensible next day in `YYYY-MM-DD`). Wire add/remove of dates and slots through `emit('update:modelValue', …)` (immutably) so the view owns no mutation logic. `sortOrder` may be omitted on POST (backend treats slots/dates positionally / defaults) — **skip drag-to-reorder** this phase; rely on array order.
7. **pollStore.create** — in `frontend/src/stores/pollStore.ts`, add `async create(payload)` that calls `apiFetch('/polls', { method: 'POST', body: JSON.stringify(payload) })` (the Phase 2 client already sets `credentials: "include"` + JSON headers), parses the `CreatedPoll`, and returns it. On non-2xx, surface the server message; specifically catch `409` (duplicate email/slot conflict — unlikely on create but handled centrally) and class-validator `400` (show first message). Keep `creating`/`error` refs. **Note for later phases:** persisting `participantToken` in `localStorage` keyed by `publicToken` belongs to the public-submit flow (Phase 5/6), NOT here — create returns no participantToken; do not add it now.
8. **PollEditor view** — in `frontend/src/views/PollEditor.vue`, lay out the two-column grid from `poll-editor.html` (`lg:grid-cols-[minmax(0,1fr)_360px]`): LEFT = a `bg-surface` card with title input, description textarea, `TimezoneSelect`, and a "Responses close · optional" field, then `<DateSlotEditor v-model="dates" :timezone="timezone" />`; RIGHT = the sticky **Preview** aside (title/desc/Open badge, timezone + humanized closesAt via `formatCloseLabel`, and a STATIC mock constellation of slot rows — illustration only, do **not** fetch tallies). Header shows breadcrumb "Polls / New poll", `h1` "Create a poll", subheading "Find the time everyone can make. Takes about a minute." Initialize `timezone = defaultTimezone()`, `dates = [oneDefaultDate]`.
9. **closesAt handling** — keep a local `closesAtIso` (or `null`) in the view from the "Responses close" field. Because `closesAt` is PATCH-only, on submit: POST the poll via `pollStore.create()`, then if `closesAtIso` is set, leave the follow-up `PATCH /api/polls/:id` to be wired when `pollStore.update()` lands (later phase) — for THIS phase, surface the field in the UI and store it, but only POST create-eligible fields. Add a short code comment marking the PATCH follow-up TODO so the next phase can pick it up.
10. **Validate before POST** — block submit and show inline `text-coral` errors when: title empty (`A title is needed before you can share this poll.`), zero dates, any date with zero slots, or any non-all-day slot missing start or end. Only when valid, call `create()`.
11. **On success** — capture `id`/`publicToken`/`shareUrl` from the response and `router.push({ name: 'poll-manage', params: { id } })` (the `/polls/:id` manage view; if that route name isn't registered yet from a sibling phase, push the path `'/polls/' + id`). Reset `creating` and clear `error`.
12. **Router** — register the `/polls/new` route in `frontend/src/router/index.ts` (name `poll-new`, `meta.requiresAuth = true`, lazy import). Do not re-implement the guard — Phase 2 owns it and redirects unauthenticated users to `/dashboard`.
13. **Tests** — write `frontend/src/components/__tests__/DateSlotEditor.spec.ts` (see Verification) covering add/remove date, add/remove slot, the all-day toggle suppressing time inputs, and the reactive count badge.

## Verification
- `cd frontend && npm run build` (or `npx vue-tsc --noEmit`) passes with TypeScript strict — confirm the exact typecheck/lint script names from the create-vue scaffold's `package.json` (`npm run type-check`, `npm run lint`) and run them.
- `cd frontend && npm run test:unit` (Vitest, scaffolded by create-vue) green for `DateSlotEditor.spec.ts`: (a) "+ Add date" grows the emitted `modelValue` by one date with one default slot; (b) "+ Add slot" grows a date's slots and the badge shows the new totals; (c) removing the last slot or a date emits the reduced array; (d) toggling a slot to all-day hides the start/end `type="time"` inputs and emits `isAllDay: true` with no times.
- Manual UI check against `docs/design/mockups/screens/poll-editor.html` and `docs/design/mockups/components/date-slot-editor.html`: visiting `/polls/new` while signed in renders the two-column editor with the Dusk theme; numerals (day chip "26", times "18:00") use the display font; the sticky preview reflects live title/desc/timezone; the "N dates · M slots" badge updates as dates/slots change.
- Manual API check: with the backend running and a valid auth cookie, filling a title + one date + one time-range slot and clicking "Create poll" issues `POST /api/polls` and returns 201 `{ id, publicToken, shareUrl, title, status: 'open' }`; the app navigates to `/polls/:id`. Submitting with an empty title or a date with no slots is blocked client-side with coral error text. (Confirm dev share links/poll appear; the magic-link/Mailpit flow is Phase 2's concern.)
