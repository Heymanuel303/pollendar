# Phase 7: End-to-end wiring & production hardening

**Plan:** [frontend-build-out](00-overview.md)
**Depends on:** 06-poll-manage-results.md
**Execution:** solo

## Context
Pollendar's frontend is a Vue 3.5 + Vite + TypeScript SPA that lets a creator find the time everyone can make: sign in by magic link, build a poll, share it, collect tri-state availability, and watch the best ("in bloom") slot emerge before completing the poll. By the end of phase 6 every route/view/component from DESIGN.md §8 is built and styled in the "Dusk Calendar" dark theme (Tailwind v4 CSS-first: `@import "tailwindcss"` + the `@theme {}` token block in `src/assets/main.css`, no `tailwind.config.js`, no PostCSS). What is still missing is the cross-flow proof that all the pieces talk to the live NestJS backend at `/api` with cookie credentials, the build-time configuration that lets the same bundle point at dev vs prod origins, and the backend CORS that credentialed requests require. This phase closes those gaps.

## Objective
Wire the full app together end-to-end with a Playwright happy-path test (sign in → create → share → vote → best updates → complete → email lands in Mailpit), introduce `VITE_`-prefixed build-time config for the API base URL and app origin, enable credentialed CORS on the backend, and produce a clean passing production build.

## Steps
1. **Enable CORS on the backend.** In `backend/src/main.ts`, after `app.get(ConfigService)`, read `configService.get<string>('CORS_ORIGINS')`, split on `,`, trim, and call `app.enableCors({ origin: <that array>, credentials: true })`. Without `credentials: true` plus an explicit origin list (not `*`), the browser drops the `Set-Cookie` from `/auth/verify` and every subsequent `credentials: "include"` request is anonymous. Keep this the only backend edit.
2. **Add frontend env config.** Create `frontend/.env.example` with `VITE_API_BASE_URL=http://localhost:3000/api` and `VITE_APP_URL=http://localhost:5173`; copy to `frontend/.env`. Add the `ImportMetaEnv` augmentation so `import.meta.env.VITE_API_BASE_URL` typechecks under `strict`. Note in the file that production overrides via `.env.production` (e.g. `VITE_API_BASE_URL=https://api.pollendar.example/api`, `VITE_APP_URL=https://pollendar.example`).
3. **Route the fetch client through env.** In the shared HTTP client, set the base to `import.meta.env.VITE_API_BASE_URL` and assert `credentials: "include"` is present on every call (auth depends on the httpOnly access/refresh cookies; the SPA never reads the JWT). Verify the 401→refresh→retry path (if present from the auth phase) still uses the env base.
4. **Route share links through env/origin.** In the `ShareBox` link util, build `${import.meta.env.VITE_APP_URL ?? window.location.origin}/p/${publicToken}`. The backend's `shareUrl` field is built server-side from `APP_URL`; the client must still be able to render a correct link from the app origin and must not hard-code `localhost`.
5. **Install & configure Playwright.** `npm i -D @playwright/test` in `frontend/`, then `npx playwright install chromium`. Write `playwright.config.ts` with `use.baseURL = process.env.VITE_APP_URL ?? 'http://localhost:5173'`, a `webServer` block (`command: 'npm run dev'`, `url: baseURL`, `reuseExistingServer: !process.env.CI`), and a single `chromium` project. Document the prerequisite in the README: the docker-compose stack (MySQL + Mailpit) and the NestJS backend on `:3000` must be running, and the DB seeded/migrated, before `test:e2e`.
6. **Write the Mailpit helper.** `frontend/e2e/helpers/mailpit.ts`: `getLatestMessageTo(email)` polls `GET ${MAILPIT_URL}/api/v1/search?query=to:<email>` (or `/api/v1/messages`) until a message appears, then `GET /api/v1/message/<ID>` to fetch the HTML/text body. Export `extractMagicLinkToken(body)` that pulls the `?token=` value from the magic-link URL (the link points at `${APP_URL}/auth/callback?token=...`), and `assertContains(body, substring)` for the completion email. Add a small retry/backoff (Mailpit delivery is async).
7. **Write the happy-path spec** (`frontend/e2e/happy-path.spec.ts`), using a unique throwaway email per run (e.g. `creator+${Date.now()}@example.test`) to avoid the `@@unique([pollId, email])` 409 and the 5-req/60s magic-link throttle:
   1. **Sign in.** Visit `/`, enter the creator email, submit. Assert the "check your inbox" / "We'll email you a magic link." copy renders (UI shows it regardless — anti-enumeration). Poll Mailpit for the message to that address, extract the token, navigate to `/auth/callback?token=<token>`. The callback view POSTs `/api/auth/verify`, cookies are set, and the app redirects to `/dashboard`. Assert we land on `/dashboard`.
   2. **Create.** Go to `/polls/new`, fill title (e.g. "Team sync") + a description, keep/confirm the timezone, add one date with ≥1 slot (the `DateSlotEditor` must block submit if any date has zero slots — `CreatePollDto` requires `dates[].slots` ArrayMinSize(1)). Submit → `POST /api/polls` returns `201 { id, publicToken, shareUrl, ... }`. Assert redirect to `/polls/:id` (manage view).
   3. **Share.** On the manage view, read the share link from `ShareBox`; assert it equals `${VITE_APP_URL}/p/<publicToken>` and capture `publicToken` for the next step.
   4. **Vote** (second browser context so it's a fresh, un-authed participant). Open `/p/<publicToken>`, assert the poll timezone is shown ("This poll's times are in {timezone}."), set a `displayName`, cycle each slot's `AvailabilityToggle` to `available`/`maybe`/`unavailable` (UI dot colors yes/maybe/no, but the payload sends the server enum strings `available`/`maybe`/`unavailable`), submit. `POST /api/public/polls/:publicToken/responses` returns `201 { publicToken }`; the app navigates to `/p/<publicToken>/done` and persists the `participantToken` in localStorage keyed by `publicToken`. Assert the thanks copy ("We've let everyone know." / done-screen text per `public-thanks.html`).
   5. **Best updates.** Back in the creator context, reload `/polls/:id`; the manage view's `pollStore`/`publicPollStore` calls `GET /api/public/polls/:publicToken/results`. Assert the `BestSlotBadge` now shows the in-bloom slot (the slot the participant marked `available`) with the `score`, and the `ResultsTable` constellation shows the participant's answer. (Best is computed fresh each fetch; with one participant the best is non-null.)
   6. **Complete.** Click complete, choose the best slot as `finalSlotId`; `POST /api/polls/:id/complete` returns the poll with `status: "completed"`. Assert the UI flips to the completed/"Closed." state. Because `email_log` is `@@unique([pollId, participantId, type])`, this fires the completion email exactly once.
   7. **Email in Mailpit.** Poll Mailpit for the completion email sent to the participant (or creator, per the completion-notifications phase); assert the body references the poll title / final time. Re-completing the same final slot must NOT resend (idempotent), so do not assert a second email.
8. **Throttle/flake guards.** In the spec, keep magic-link requests to one per run and use unique emails so reruns don't trip the per-IP/per-email rate limit (5/60s). Wrap Mailpit reads in the helper's backoff. Handle the invalid/expired-token branch only as a negative note in the README (not a happy-path assertion).
9. **Production build.** Run `npm run build` in `frontend/`; fix any `vue-tsc` strict errors surfaced by the new `import.meta.env` types or the env augmentation. Confirm the bundle reads `VITE_API_BASE_URL`/`VITE_APP_URL` at build time (a `.env.production` value bakes into `dist/`).
10. **Write production cookie/CORS notes** in `frontend/README.md`: in prod the backend needs `COOKIE_SECURE=true` (cookies only over HTTPS), `COOKIE_DOMAIN` set to the real apex/sub domain (not `localhost`), and `CORS_ORIGINS` set to the prod frontend origin; the SPA must be served from an HTTPS origin that matches a `CORS_ORIGINS` entry, with `SameSite=Lax` cookies working because frontend and API share the registrable domain (note the cross-site caveat + `SameSite=None; Secure` if they are split across sites). State that `.env.production` must set `VITE_API_BASE_URL`/`VITE_APP_URL` to the prod URLs. (Boot-time validation of these prod env vars is phase 8's job.)
