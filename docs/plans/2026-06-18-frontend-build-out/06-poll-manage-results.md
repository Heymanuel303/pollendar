# Phase 6: Poll manage view — results, best slot, share & complete flow

**Plan:** [frontend-build-out](00-overview.md)
**Depends on:** 05-public-poll-flow.md
**Execution:** workflow

## Context
We are building the real Pollendar frontend — a Vue 3.5 + Vite + TypeScript SPA implementing every route/view in DESIGN.md §8, styled to the "Dusk Calendar" dark theme and wired to the finished NestJS backend at `/api` with cookie credentials. Phase 5 delivered the public participant flow (`PublicPoll` + `publicPollStore`); this phase delivers the creator-facing `/polls/:id` view — `PollManage` — where the owner sees who responded, which slot is in bloom, copies the share link/invite message, and completes the poll. All styling uses Tailwind v4 CSS-first only (`@import "tailwindcss";` + the `@theme {}` token block from DESIGN-UI.md, established in the scaffold phase); the mockup motif helpers (`.bg-dusk`, `.pollen-dot`, `.bloom`, `.bloom-bg`) are the small custom classes already in `src/assets/main.css` — components reference them, they are not re-declared here.

## Objective
Build the thin `PollManage` view that composes five independent components — read-only `AvailabilityGrid` (constellation matrix), `ResultsTable` (per-slot tallies), `BestSlotBloom` + `BestSlotBadge` (the blooming winner), and `ShareBox` (copy link + copy invite message) — plus the confirm-and-complete flow, backed by `pollStore.get(id)` + `pollStore.complete(pollId, finalSlotId)` and the live public results endpoint.

### Backend wire shapes (do not invent — these are the real responses)
- `GET /api/polls/:id` (auth cookie; 404 if not owned/missing): `{ id: string, publicToken: string, title: string, description: string|null, timezone: string, status: 'open'|'completed'|'cancelled', closesAt: string|null, completedAt: string|null, finalSlotId: string|null, dates: { id, eventDate: 'YYYY-MM-DD', sortOrder, slots: { id, startTime: 'HH:mm:ss'|null, endTime: string|null, isAllDay: boolean, label: string|null, sortOrder }[] }[] }`. NOTE: this endpoint does NOT include per-participant responses; the constellation matrix data comes from the results endpoint, so `AvailabilityGrid` renders from aggregate tallies, not raw rows (see step 4).
- `GET /api/public/polls/:publicToken/results` (no auth): `{ best: { slotId, date: 'YYYY-MM-DD', label: string|null, score } | null, slots: { slotId, available, maybe, unavailable, score }[] }`. Computed LIVE; `slots[]` order is NOT result-sorted — join to `poll.dates[].slots[]` by `slotId` for display order; `best.slotId` identifies the bloom. `best` is `null` only when the poll has zero slots.
- `POST /api/polls/:id/complete` (auth cookie + ownership) body `{ finalSlotId: string }` (numeric string): 200 returns the updated poll; 400 if the slot is not in this poll; 409 if poll not `open`; idempotent on an already-completed poll (no re-notify). Only show the Complete button when `status === 'open'`.
- `GET /api/polls/:id/invite-message` (auth cookie + ownership): `{ message: string, shareUrl: string }`. The `message` is the minimal backend template (`You're invited to vote on "{title}". Pick your availability here: {shareUrl}`); the frontend renders the fuller DESIGN.md §7 text (title, description, `Add your availability here (takes ~1 min):`, `shareUrl`, and `Please reply before {closesAtHuman}.` when `closesAt` is set). Build `shareUrl` from `window.location.origin` + `/p/${publicToken}` (or the `VITE_` app-origin var) — use the endpoint's `shareUrl` as the canonical value but never hard-code localhost.

## Steps
1. **pollStore extensions** — in `frontend/src/stores/pollStore.ts`, add async actions using the shared fetch client (all `credentials:"include"`):
   - `get(id)` → GET `/api/polls/:id`; store as `currentPoll`. On 404 set an `error` ("Poll not found") the view renders as an empty state.
   - `loadResults(publicToken)` → GET `/api/public/polls/:publicToken/results`; store as `results` ({ best, slots }).
   - `loadInviteMessage(id)` → GET `/api/polls/:id/invite-message`; keep `{ message, shareUrl }` so `ShareBox` has the canonical `shareUrl` + can build the full message.
   - `complete(pollId, finalSlotId)` → POST `/api/polls/:id/complete` with `{ finalSlotId }`; on 200 replace `currentPoll` with the returned updated poll (now `status:'completed'`, `finalSlotId`, `completedAt`). Surface 409 cleanly ("Poll is no longer open") and 400 ("That slot isn't part of this poll"). Re-fetch results after completion so the bloom reflects final state.
   Keep these as discrete actions with `loading`/`error` refs; the view only calls actions and reads refs.
2. **Shared types + format helpers** — confirm `OwnedPoll`, `PollResults`, `SlotTallyResult`, `BestSlot` in `frontend/src/lib/types.ts` (ids `string`; `availability` union `'available'|'maybe'|'unavailable'` — backend strings, NOT yes/maybe/no). In `formatTime.ts` ensure helpers exist for: a short date label (`Thu Jun 26`), a slot time-range/all-day label (`18:00–20:00` / `All day` / `label`), and a human `closesAt` (`Thu Jun 25, 18:00`) — all via `Intl.DateTimeFormat` with `{ timeZone: poll.timezone }`. Numerals shown via the `.num`/`font-display` utility per DESIGN-UI.
3. **`ui/CopyButton.vue`** — props `{ text: string, label?: string, copiedLabel?: string, variant?: 'pollen'|'ghost' }`; on click `await navigator.clipboard.writeText(text)`, set `copied=true`, `setTimeout(()=>copied=false, 2000)`. Copied state = mint styling (`border-mint/40 bg-mint/15 text-mint ring-1 ring-mint/30`, check icon, `Copied ✓`) per the share-box mockup "Copied state".
4. **`AvailabilityGrid.vue`** — props `{ dates: OwnedPoll['dates'], results: PollResults }`. Render the read-only constellation matrix per `components/availability-grid.html`: grouped date header row + slot sub-header row; sticky `Who`/first column; each slot column shows its aggregate as dot clusters — render `available` count as `.pollen-dot`, `maybe` as ring-outline dots, `unavailable` as `bg-no` dots (the per-participant identity is not exposed by the API, so the matrix visualizes the tally distribution as a constellation, matching the mockup's dot density). Apply `.bloom-bg` to the column whose `slotId === results.best?.slotId`. Legend (Yes/Maybe/No) at top. Horizontal scroll wrapper (`overflow-x-auto`, `min-w-[640px]`). Use the color-naming map: yes=pollen, maybe=#B8B3DE, no=#3A3563.
5. **`ResultsTable.vue`** — props `{ slots: OwnedPoll dates flattened to slot meta, results: PollResults['slots'], best: BestSlot|null }`. For each slot (in poll display order, joined by `slotId`): slot date + label/time, a distribution bar (`flex h-2.5 rounded-full bg-no/60` with `bg-yes` then `bg-maybe/70` spans at `width: pct%` where pct = count/total), numeric `N yes · N maybe · N no`, and the score in `font-display`. Winner row (`slotId === best?.slotId`) gets `bloom-bg ring-1 ring-pollen/40` + `✦ In bloom` pill and `.bloom` glow. Footer: `Score = 2 × yes + maybe. The highest score blooms.` and `{N} participants · best recomputed on every response` (N = max responses across slots, i.e. `available+maybe+unavailable` of the most-answered slot).
6. **`BestSlotBloom.vue`** — props `{ best: BestSlot|null, slot: slot meta, timezone, status, responseCount, canComplete: boolean }`; emit `complete`. Render the large bloom card (`.bloom .bloom-bg border-pollen/40`): `✦ In bloom` badge, big `font-display` score, `{weekday, date}` + `{time range · label}` (poll tz), a pollen-dot cluster sized to `best` available count + `{a} of {n} available`, timezone string. When `canComplete` (status==='open'), show the `✦ Complete poll` button (`bg-pollen shadow-glow`) + `Confirm {date · label} as the final time`; when completed, swap the button area for a `Completed {completedAt}` state. Emits `complete` (parent owns the confirm dialog + store call). Render nothing / an empty hint when `best === null` (zero responses).
7. **`BestSlotBadge.vue`** — props `{ best: BestSlot|null, slot: slot meta, timezone }`. Compact: `✦ In bloom` pill + `Best: {date · label}` chip with a `pollen-dot`. Renders nothing when `best===null`. Used by `ShareBox` footer and the completed-poll header.
8. **`ShareBox.vue`** — props `{ shareUrl: string, poll: OwnedPoll, best: BestSlot|null, bestSlot: slot meta }`. Computed `inviteMessage` builds the full DESIGN.md §7 template from `poll.title`, `poll.description`, `shareUrl`, and `closesAt` (human via `formatTime`). Layout per `components/share-box.html`: status pill, `Public link` readonly input + `<CopyButton :text="shareUrl" label="Copy link" variant="pollen"/>`, `Invite message` readonly `whitespace-pre-line` preview + `<CopyButton :text="inviteMessage" label="Copy invite message" variant="ghost"/>`, footer `<BestSlotBadge/>` note. No "Edit" button this phase.
9. **`PollManage.vue` (thin view)** — on mount: `const id = route.params.id as string`; `await pollStore.get(id)`; then `pollStore.loadResults(currentPoll.publicToken)` + `pollStore.loadInviteMessage(id)`. Template: breadcrumb `← Polls` (router-link to `/dashboard`), header (title, status pill `Open · gathering responses` / `Completed`, `Timezone {poll.timezone}`, `Responses close {closesAtHuman}`), then `BestSlotBloom`, a two-column body with `AvailabilityGrid` + `ResultsTable` (left, `lg:col-span-2`) and a right column with a participants summary card (responded count from results, email-notify note) + `ShareBox`. Listen for `@complete` from `BestSlotBloom` → open a confirm dialog ("Confirm {date · label} as the final time? Participants who left an email will be notified.") → on confirm call `pollStore.complete(id, best.slotId)`; on success the store updates `currentPoll.status` and the bloom switches to completed. Show loading + 404/error states from the store. Keep zero data-fetching in component setup beyond delegating to store actions.
10. **Wire-up sanity** — `PollManage` derives a `slotMetaById` map (from `currentPoll.dates[].slots[]`) once and passes the relevant slot meta into each child so `ResultsTable`/`BestSlotBloom`/`BestSlotBadge` never re-walk the tree. `finalSlotId`/`completedAt` are shown ONLY here (creator view), never leaked to public components.

## Execution strategy
Because Execution is `workflow`:
- **Fan-out unit:** one agent per independent component — `AvailabilityGrid.vue`, `ResultsTable.vue`, `BestSlotBloom.vue` + `BestSlotBadge.vue` (one agent, sibling files), `ShareBox.vue` + `ui/CopyButton.vue` (one agent). Each edits distinct files and can be built in parallel against the documented `OwnedPoll`/`PollResults` prop contracts in step 0. The shared `pollStore` + `types.ts` + `formatTime.ts` changes (steps 1–2) are a prerequisite stub the lead does FIRST so every component imports stable types; the `PollManage.vue` view assembly (steps 9–10) is done AFTER the components land.
- **Shape:** pipeline — (a) extend store/types/format helpers, (b) parallel barrier building the four component units against those contracts, (c) assemble `PollManage.vue`, (d) verify pass. The view assembly genuinely needs all components present, so step (b) is a true barrier before (c).
- **Isolation:** none — every agent writes a distinct `.vue`/`.ts` file in the shared `frontend/src` tree; no worktrees. The store/types step and the view-assembly step are ordering dependencies, not file conflicts.
- **Verify stage:** `vue-tsc --noEmit` typechecks the whole tree; each component is rendered against the documented `PollResults` shape (a fixture with 4 slots, one clear winner) to confirm it compiles + the winner blooms in EXACTLY one place per component (one bloom column in the grid, one bloom row in the table, one bloom card, one badge). Confirm completed-state hides the Complete button.
