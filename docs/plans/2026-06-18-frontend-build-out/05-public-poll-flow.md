# Phase 5: Public participant flow (PublicPoll + AvailabilityToggle + PublicThanks)

**Plan:** [frontend-build-out](00-overview.md)
**Depends on:** 04-dashboard.md
**Execution:** solo

## Context
We are building the real Pollendar frontend — a Vue 3.5 + Vite + TypeScript SPA wired to the finished NestJS backend at `/api` with cookie credentials — implementing the Dusk Calendar design system (`docs/design/DESIGN-UI.md`). This phase delivers the **anonymous participant flow**: a person who opens a share link `/p/:publicToken`, marks their availability per slot, gives a name (email optional), submits, and lands on a thank-you page showing the current best slot plus share buttons. It is the only part of the app reachable without auth, so it has its own minimal header (wordmark only, no app nav). All styling uses the Dusk Calendar tokens and the Tailwind v4 **CSS-first** setup established in phase 1 (`src/assets/main.css` with the `@theme {}` block and the `.bg-dusk` / `.pollen-dot` / `.bloom` / `.bloom-bg` motif helpers); no `tailwind.config.js`, no PostCSS, no v3 `@tailwind` directives.

## Objective
Ship the `PublicPoll` and `PublicThanks` views, the tri-state `AvailabilityToggle` component (one per slot), and a `publicPollStore` (`load` / `submit` / `results`) that persists the participant's edit token so they can later edit, with all slot times rendered in the poll's IANA timezone.

## Backend contract (the real, implemented shape — authoritative)
Read directly from `backend/src/public/`:
- `GET /api/public/polls/:publicToken` → `PublicPoll` `{ id: string, title, description: string|null, timezone: string, status: PollStatus, dates: [{ id: string, eventDate: ISO-date, sortOrder: number, slots: [{ id: string, startTime: ISO-datetime|null, endTime: ISO-datetime|null, isAllDay: boolean, label: string|null, sortOrder: number }] }] }`. 404 on unknown token. No participant data leaks.
- `POST /api/public/polls/:publicToken/responses` → **201 `{ publicToken: string }`** — and **nothing else**. This `publicToken` is the *participant's own* opaque edit token (the `participants.public_token`, distinct from the poll's share token). Body is `SubmitResponsesDto`: `{ displayName: string (≤120), email?: string (valid, ≤255), answers: [{ pollSlotId: string, availability: 'available'|'maybe'|'unavailable' }] (≥1) }`. 409 on duplicate email per poll with message `"A participant with this email already responded to this poll"`; 400 on invalid/foreign slot id or missing fields.
- `GET /api/public/polls/:publicToken/results` → `PollResults` `{ best: { slotId: string, date: 'YYYY-MM-DD', label: string|null, score: number } | null, slots: [{ slotId: string, available: number, maybe: number, unavailable: number, score: number }] }`. Live computation; 404 on unknown token. `best` is `null` only when the poll has zero slots.
- **`submit` does NOT return results** (the §6 DESIGN.md example showing `{ participantToken, results }` is stale — `public.service.ts` returns only `{ publicToken }`). PublicThanks must call `GET /results` separately. Use the field name from the wire (`publicToken`); internally treat it as the participant edit token.
- `PUT /api/public/participants/:participantToken` (the participant edit path from DESIGN.md §5) is **NOT implemented on the backend** — `PublicController` has only `GET /polls/:token`, `GET /polls/:token/results`, and `POST /polls/:token/responses`. So the "edit my response" flow cannot be fully wired in this phase. Still **persist** the returned `publicToken` (the participant's edit token) to localStorage now so a future backend phase that adds the PUT route can light up editing without a data-loss; but the "Edit my response" link should point back to `/p/:publicToken` (re-open the form) rather than a non-existent edit endpoint. Flag this gap (see the overview's Open questions).

## Files to touch
- `frontend/src/lib/api/public-poll.ts` — typed fetch wrappers + DTO/response types for the three public endpoints; all calls go through the shared client with `credentials: "include"`. (Public endpoints need no cookie, but the shared client adds it uniformly — harmless here.)
- `frontend/src/lib/participantToken.ts` — tiny localStorage helper: persist/read the participant's edit token keyed by the poll's `publicToken`.
- `frontend/src/stores/publicPollStore.ts` — Pinia store: `load(token)`, `submit(payload)`, `results(token)`, plus reactive `poll`, `results`, `status`/`error` flags. Data-fetching lives HERE, not in the views.
- `frontend/src/components/AvailabilityToggle.vue` — tri-state Yes/Maybe/No button group, `v-model`-style (`modelValue: Availability | null`, emits `update:modelValue`); cycles yes→maybe→no→none on re-tap; colors per mockup.
- `frontend/src/components/PollSlotRow.vue` — a single slot row (label + time range in poll tz + optional "✦ In bloom" + an `AvailabilityToggle`). Keeps `PublicPoll` thin. (Small focused component; do NOT inline this into the view.)
- `frontend/src/views/PublicPoll.vue` — composes the date groups → `PollSlotRow` list, the "About you" form, the sticky "Leaning so far" footer with Submit. Thin: delegates fetch/submit to the store.
- `frontend/src/views/PublicThanks.vue` — bloom hero, best-slot card (from `results`), Share buttons (copy link / copy invite), "Edit my response" link.
- `frontend/src/components/ui/CopyButton.vue` — (only if phase 3/4 did not already create a copy-to-clipboard primitive) small button that copies a string and shows a transient "Copied" state. Reuse the existing one if present.
- `frontend/src/lib/datetime.ts` — extend (or create if phase 1/3 did not) the shared `Intl.DateTimeFormat`-based slot formatter used to render times in `poll.timezone`. Reuse, do not duplicate.
- `frontend/src/router/index.ts` — confirm/add the two public routes `/p/:publicToken` → `PublicPoll` and `/p/:publicToken/done` → `PublicThanks`, both `meta: { requiresAuth: false }` (or absence of the auth guard), matching DESIGN §8.
- Tests (see Verification): `frontend/src/components/__tests__/AvailabilityToggle.spec.ts`, `frontend/src/stores/__tests__/publicPollStore.spec.ts`.

## Steps
1. **API layer** — in `frontend/src/lib/api/public-poll.ts`, define the wire types mirroring the backend DTOs verbatim: `Availability = 'available' | 'maybe' | 'unavailable'`; `PublicPoll`, `PublicPollDate`, `PublicPollSlot` (all `id` fields typed `string`, `startTime`/`endTime` as `string | null`); `SubmitResponsesPayload` (`displayName`, optional `email`, `answers: { pollSlotId: string; availability: Availability }[]`); `SubmitResponse = { publicToken: string }`; `PollResults` / `BestSlot` / `SlotTally`. Export `getPublicPoll(token)`, `submitResponses(token, payload)`, `getResults(token)` that call the shared fetch client (from phase 2) against `/public/polls/:token`, `/public/polls/:token/responses` (POST), `/public/polls/:token/results`. The client throws a typed `ApiError` carrying `status` + `message`; surface 409/404/400 codes upward.
2. **Participant-token persistence** — in `frontend/src/lib/participantToken.ts`, implement `saveParticipantToken(pollPublicToken: string, participantToken: string)`, `getParticipantToken(pollPublicToken: string): string | null`, `clearParticipantToken(pollPublicToken)`. Use a namespaced key, e.g. `pollendar:pt:<pollPublicToken>`. Wrap `localStorage` access in try/catch (private-mode / disabled storage must not crash the flow).
3. **Store** — in `frontend/src/stores/publicPollStore.ts`, a Pinia setup-store with refs `poll: PublicPoll | null`, `results: PollResults | null`, `loadState`/`submitState` (`'idle'|'loading'|'success'|'error'`), `errorCode: number | null`, `errorMessage: string | null`. Actions: `load(token)` → `getPublicPoll`; `submit(token, payload)` → `submitResponses`, on success `saveParticipantToken(token, res.publicToken)` and return the token; `results(token)` → `getResults`. On a 409 from `submit`, set `errorCode = 409` + the server message and do NOT navigate. Keep all `try/catch`/state-transition logic here so views stay declarative.
4. **AvailabilityToggle** — in `frontend/src/components/AvailabilityToggle.vue`, render the three-button group from `docs/design/mockups/components/availability-toggle.html`: wrapper `inline-flex rounded-xl border border-line bg-canvas p-1 text-sm font-medium`. Props `modelValue: Availability | null`. Each button maps a UI label→backend value: **Yes→`available`**, **Maybe→`maybe`**, **No→`unavailable`** (never send yes/maybe/no to the API). Active styling per mockup: Yes active `bg-yes text-canvas shadow-glow`; Maybe active `bg-maybe text-canvas`; No active `bg-no text-moonlight`; inactive `text-dim hover:text-moonlight focus:ring-2 focus:ring-pollen/40`. Clicking the active button again clears to `null` (cycle to "none"). Emit `update:modelValue`. Buttons are real `<button type="button">` with `aria-pressed`; the group has an accessible label.
5. **PollSlotRow** — in `frontend/src/components/PollSlotRow.vue`, props `slot: PublicPollSlot`, `eventDate: string`, `timezone: string`, `isBest: boolean`, `modelValue`. Render label (or fallback) + the time line: when `slot.isAllDay` show "Any time works"; otherwise format `startTime`–`endTime` as `HH:MM–HH:MM` via `lib/datetime.ts` using `Intl.DateTimeFormat(locale, { hour: '2-digit', minute: '2-digit', hour12: false, timeZone: poll.timezone })`. When `isBest`, apply the bloom treatment from the mockup (`bloom-bg border-pollen/40 shadow-glow` + a `pollen-dot` + the `✦ In bloom` badge). Embed `<AvailabilityToggle>` bound to `modelValue`. Wrap numerals in the `.num` (Space Grotesk) utility per DESIGN-UI.
6. **PublicPoll view** — in `frontend/src/views/PublicPoll.vue`, read `route.params.publicToken`, call `store.load(token)` in `onMounted` (NOT inline fetch). Local reactive state: `answers = reactive<Record<slotId, Availability | null>>` and a `form = reactive({ displayName: '', email: '' })`. Render, per `docs/design/mockups/screens/public-poll.html`: the public header (wordmark + `pollendar.app`); poll intro (status pill, title, description, the timezone badge "Times shown in {poll.timezone}", and a "Responses close …" badge **only if** the poll exposes `closesAt` — guard since the public DTO may omit it); the "Your availability" card with one date group per `poll.dates`, each containing `PollSlotRow`s (mark the row whose `slot.id === results?.best?.slotId` as `isBest`, after results load — see step 7 for when results are fetched on this view, or simply compute "leaning so far" client-side from current answers if results aren't loaded yet); the "About you" grid (`sm:grid-cols-2`) with required name (`maxlength=120`) and optional email (`type=email maxlength=255`); and the sticky footer ("Leaning so far" + current best label + Submit button `bg-pollen shadow-glow hover:brightness-110`). Disable Submit while `displayName` is empty or no answer chosen. Form fields use the mockup classes (`bg-canvas border-line placeholder:text-mute focus:border-pollen focus:ring-2 focus:ring-pollen/30`).
7. **Submit + navigation** — on Submit, build `answers[]` from the `answers` map (include only slots the participant actually set; require `≥1`), call `store.submit(token, payload)`. On success, route to `/p/:publicToken/done`. On a **409** (duplicate email), do NOT navigate: show a coral inline alert near the Submit button using `store.errorMessage` (the server's exact message), leaving the form intact so the participant can change their email. On 400/network error, show a generic inline error. Only block status: if `poll.status !== 'open'`, render a read-only "This poll is closed" notice instead of the input + sticky footer (polls are editable/answerable only while open).
8. **PublicThanks view** — in `frontend/src/views/PublicThanks.vue`, on mount call `store.load(token)` (for title/timezone if arriving cold via the share URL) and `store.results(token)`, per `docs/design/mockups/screens/public-thanks.html`. Render: bloom hero (`✦` centerpiece in `pollen/15` + `bloom-bg` + `shadow-glow`), "You're in!" + name echo (pass the just-submitted name via router state/query or read from the form store — if unavailable on cold load, omit the name gracefully); the best-slot bloom card built from `results.best` (label + date formatted in poll tz + `score`/yes count), with the "✦ In bloom" badge and the "This can still change as more people respond." note — handle `results.best === null` by hiding the card; the Share section with a truncated read-only URL field (built from `window.location.origin` or `VITE_APP_URL`, never hard-coded localhost: `${origin}/p/${publicToken}`), a **Copy link** button (`bg-pollen` fill, copies the share URL) and a **Copy invite** button (secondary `border-line`, copies a short invite message containing the URL); and an "Edit my response" link shown only when `getParticipantToken(token)` exists (links toward the future edit flow / back to `/p/:publicToken`).
9. **Share/copy** — reuse the existing clipboard primitive if phase 3/4 created `components/ui/CopyButton.vue`; otherwise add it (copies a passed string via `navigator.clipboard.writeText`, falls back to a hidden `<textarea>` + `execCommand` if unavailable, and flips label to "Copied" for ~1.5s). Build the invite message text in a small pure helper so it is testable.
10. **Router** — in `frontend/src/router/index.ts`, ensure both routes exist and are excluded from the auth guard (public). Lazy-load both views. Confirm the `/done` suffix matches DESIGN §8 (`/p/:publicToken/done`, not `/thanks`).
11. **Motif helpers** — verify `.bg-dusk`, `.pollen-dot`, `.bloom`, `.bloom-bg` already exist in `src/assets/main.css` from phase 1; if any are missing, port them verbatim from the mockup `<style>` blocks (do not reinvent the palette). Do NOT add a `tailwind.config.js`.
