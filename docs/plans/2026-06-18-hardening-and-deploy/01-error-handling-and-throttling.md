# Phase 1: Global exception filter, public-submission throttle, and prod cookie/anti-enumeration audit

**Plan:** [hardening-and-deploy](00-overview.md)
**Depends on:** none
**Execution:** solo

## Context
This plan delivers production-readiness basics for the Pollendar NestJS API: centralized Prisma-error -> HTTP mapping, throttling on public response submissions, audited prod cookie/anti-enumeration flags, real e2e happy paths, and deploy notes. This phase is the security/robustness slice of that goal. Already done (do NOT redo): `main.ts` wires `setGlobalPrefix('api')`, `cookieParser()`, the global `BigIntSerializerInterceptor`, a global `ValidationPipe({ whitelist, forbidNonWhitelisted, transform })`, and `enableCors` with a comma-split `CORS_ORIGINS` allow-list + `credentials: true`; `app.module.ts` wires `ThrottlerModule.forRootAsync` (THROTTLE_TTL*1000 / THROTTLE_LIMIT) and a global `APP_GUARD` `ThrottlerGuard`; `auth.controller.ts` already has `@Throttle` on `magic-link`/`verify`/`refresh` and returns `200 { ok: true }` from `magic-link` regardless of email existence; `cookie.util.ts` builds httpOnly `sameSite: 'lax'` cookies honoring `COOKIE_SECURE`/`COOKIE_DOMAIN`; and inline `Prisma.PrismaClientKnownRequestError` handling already exists in `polls.service.ts` (P2002/P2025), `public.service.ts` (P2002), and `notifications.service.ts` (P2002 race-retry).

## Objective
Add a global exception filter that centralizes unhandled-Prisma-error -> HTTP mapping as a safety net, throttle `POST /api/public/polls/:token/responses`, and verify (via tests + a documented checklist) the prod cookie flags and anti-enumeration behavior — without regressing the existing inline service mappings or the already-wired guards/pipe.

## Files to touch
- `backend/src/common/prisma-exception.filter.ts` — NEW. `@Catch(Prisma.PrismaClientKnownRequestError)` filter implementing `ExceptionFilter`; maps `P2002` -> 409, `P2025` -> 404, `P2003` -> 409 (FK), everything else -> 500 with a generic message. Returns a clean JSON body and never echoes `err.message`/`err.meta` (no internal leakage). Lets `HttpException` subclasses (the inline `ConflictException`/`NotFoundException`/`BadRequestException` thrown by services) pass through untouched, since it only catches the Prisma error type.
- `backend/src/main.ts` — register the filter via `app.useGlobalFilters(new PrismaExceptionFilter())` (do NOT add a second `ValidationPipe`, interceptor, or CORS call).
- `backend/src/public/public.controller.ts` — add `@Throttle({ default: { limit: N, ttl: MS } })` to the `submit` method (`@Post('polls/:token/responses')`) only; leave the two `@Get` methods on the global throttle default.
- `backend/src/common/prisma-exception.filter.spec.ts` — NEW. Unit-test the filter's code-to-status mapping and the no-leak body shape using a mocked `ArgumentsHost`/`Response`.
- `backend/src/public/public.controller.spec.ts` — EXISTS. Extend (do not rewrite) to assert the `@Throttle` metadata is present on `submit` (e.g. via `Reflect.getMetadata`/`THROTTLER_*` keys) and absent/default on the two getters.
- `docs/plans/2026-06-18-hardening-and-deploy/SECURITY-CHECKLIST.md` — NEW (this plan dir only). Short audited checklist: httpOnly + `sameSite: 'lax'` + `secure` (prod), `COOKIE_DOMAIN` host-only default, anti-enumeration on `magic-link`, no participant-email leak in public DTOs, CORS allow-list + credentials. Records the verification result, not new code.

## Steps
1. Create `backend/src/common/prisma-exception.filter.ts`: `import { ArgumentsHost, Catch, ExceptionFilter, HttpStatus } from '@nestjs/common'`, `import { Prisma } from '@prisma/client'`, and `import type { Response } from 'express'`. Decorate the class with `@Catch(Prisma.PrismaClientKnownRequestError)` and implement `catch(exception, host)`. Get the response via `host.switchToHttp().getResponse<Response>()`.
2. Inside `catch`, `switch (exception.code)`: `'P2002'` -> `{ status: HttpStatus.CONFLICT, message: 'Resource already exists' }`; `'P2025'` -> `{ status: HttpStatus.NOT_FOUND, message: 'Resource not found' }`; `'P2003'` -> `{ status: HttpStatus.CONFLICT, message: 'Operation violates a related-record constraint' }`; default -> `{ status: HttpStatus.INTERNAL_SERVER_ERROR, message: 'Internal server error' }`. Send `res.status(status).json({ statusCode: status, message })`. Do NOT include `exception.code`, `exception.meta`, or `exception.message` in the body. (Filter is the safety net; the existing inline `ConflictException`/`NotFoundException`/`BadRequestException` mappings in `polls.service.ts`/`public.service.ts`/`notifications.service.ts` stay — they are `HttpException`s and are not caught by `@Catch(Prisma.PrismaClientKnownRequestError)`, so behavior is preserved.)
3. In `backend/src/main.ts`, import `PrismaExceptionFilter` from `./common/prisma-exception.filter` and add `app.useGlobalFilters(new PrismaExceptionFilter());` after the existing `useGlobalInterceptors`/`useGlobalPipes` lines and before/after `enableCors` (order vs CORS is irrelevant). Touch nothing else in `main.ts`.
4. In `backend/src/public/public.controller.ts`, add `import { Throttle } from '@nestjs/throttler';` and decorate the `submit` method with `@Throttle({ default: { limit: 10, ttl: 60_000 } })` (10 submissions / 60s per IP — mirrors the auth `verify`/`refresh` shape; tune later if needed). Leave `getPoll` and `getResults` un-decorated so they keep the global default. Do NOT add `@UseGuards` — the global `APP_GUARD` `ThrottlerGuard` already enforces `@Throttle`.
5. Create `backend/src/common/prisma-exception.filter.spec.ts`: instantiate the filter, build a fake `ArgumentsHost` whose `switchToHttp().getResponse()` returns a `{ status: jest.fn().mockReturnThis(), json: jest.fn() }` stub, then for each of P2002/P2025/P2003/unknown construct a `new Prisma.PrismaClientKnownRequestError('msg', { code, clientVersion: 'x', meta: { target: ['email'] } })`, call `filter.catch(err, host)`, and assert `status` was called with the mapped code and `json` body contains the generic message and NOT the raw `meta`/`code`/`message`.
6. Extend `backend/src/public/public.controller.spec.ts` with one assertion that the `submit` handler carries throttler metadata (e.g. `Reflect.getMetadata` against the throttler metadata key on `PublicController.prototype.submit`, or assert it differs from the un-decorated `getResults`). Keep all existing tests intact.
7. Write `docs/plans/2026-06-18-hardening-and-deploy/SECURITY-CHECKLIST.md` recording the audit: confirm `baseCookieOptions` in `cookie.util.ts` sets `httpOnly: true`, `sameSite: 'lax'`, `secure` from `COOKIE_SECURE`, `domain` host-only when unset; confirm `magic-link` returns `200 { ok: true }` regardless of email; confirm `PublicService.findByPublicToken`/`getResults`/`submitResponses` never return participant emails or owner `userId`; confirm CORS is an explicit allow-list with `credentials: true`. Note `COOKIE_SECURE=true` is required in the prod `.env`.
