# Phase 2: Disposable-DB e2e harness + happy-path flows and unit gap-fill

**Plan:** [hardening-and-deploy](00-overview.md)
**Depends on:** 01-error-handling-and-throttling.md
**Execution:** workflow
**Status:** completed

## Context
Production-readiness for the Pollendar NestJS API (NestJS 11, Prisma 7 + `@prisma/adapter-mariadb` against MySQL 8.4): centralized error mapping, public-submission throttling, audited prod cookie/anti-enumeration flags, real e2e + unit gap-fill, and deploy notes. This phase owns the **test surface**: the only e2e spec today is the stock Hello-World `backend/test/app.e2e-spec.ts`, so there is no harness and no real coverage of the auth or poll lifecycles. Already done (do NOT redo): `main.ts` wires `setGlobalPrefix('api')`, `cookieParser()`, the global `BigIntSerializerInterceptor`, the global `ValidationPipe({whitelist,forbidNonWhitelisted,transform})`, and `enableCors` with a comma-split `CORS_ORIGINS` allow-list + `credentials:true`; `app.module.ts` wires `ThrottlerModule` + a global `APP_GUARD ThrottlerGuard`; `auth.controller.ts` already throttles `magic-link`/`verify`/`refresh` and `magic-link` always returns `200 { ok: true }` (anti-enumeration); `public.service.ts` already sanitizes the public poll shape (no `userId`, no participant emails). Phase 1 adds the global exception filter and the `POST polls/:token/responses` submission throttle — this phase builds tests that assume those exist.

## Objective
Stand up a disposable test-DB e2e harness and write real happy-path e2e flows (auth lifecycle + poll lifecycle), then gap-fill unit tests for the best-slot algorithm tie-break and auth token hashing/expiry.

- `backend/src/public/public.service.spec.ts` — gap-fill: add `getResults` tie-break unit tests over a mocked `$queryRaw` ordered result, asserting `best` = `rows[0]` and the 5-key ordering (score → available → unavailable → date/time → slot id) per DESIGN §4.
- `backend/src/auth/auth.service.spec.ts` — gap-fill: assert `parseTtlToMs` boundary handling (`'45s'/'15m'/'1h'/'30d'`, reject malformed), SHA-256 hex-64 stability (same input → same hash, `generateToken` ≠ stored hash), and expiry-edge rejection in `verify`/`refresh` (`expiresAt` exactly now vs. future).

## Steps
1. **Test-DB env contract.** In `backend/test/global-setup-e2e.ts`, read `process.env.TEST_DATABASE_URL`; throw if it is missing or `=== process.env.DATABASE_URL` (never point e2e at the dev DB — see plan constraints). Document the var in `.env.example` is out of scope here (deploy phase owns docs); the harness only consumes it. Set `process.env.DATABASE_URL = TEST_DATABASE_URL` so `PrismaService`'s `buildMariaDbAdapter(config.getOrThrow('DATABASE_URL'))` and the Prisma CLI both target the disposable schema.
2. **Apply migrations through the adapter path.** Still in `global-setup-e2e.ts`, shell out to `npx prisma migrate reset --force --skip-seed` (Prisma 7 reads `prisma.config.ts`, which loads the repo-root `.env`; pass the overridden `DATABASE_URL` via the child-process env). Use `--skip-seed` so the dev `tsx prisma/seed.ts` fixture is NOT loaded — e2e specs create their own data through the API. This satisfies "e2e runs against a disposable schema via prisma migrate deploy/reset".
3. **Shared bootstrap helper.** In `backend/test/setup-e2e.ts`, export `createTestApp()` that mirrors `seed.spec.ts`'s real-DB pattern but boots the **whole app**: `Test.createTestingModule({ imports: [AppModule] }).compile()`, then re-apply the production global wiring the e2e client depends on — `app.setGlobalPrefix('api')`, `app.use(cookieParser())`, `app.useGlobalInterceptors(new BigIntSerializerInterceptor())`, `app.useGlobalPipes(new ValidationPipe({ whitelist: true, forbidNonWhitelisted: true, transform: true }))` — because `createNestApplication()` does NOT replay `main.ts`'s `bootstrap()`. Return `{ app, prisma: app.get(PrismaService) }`. Add `truncateAll(prisma)` that deletes responses → participants → polls → authSession → loginToken → users (FK order) for per-flow isolation.
4. **Auth e2e (`auth.e2e-spec.ts`).** In `beforeAll` call `createTestApp()` + `app.init()`; `afterAll` `app.close()`; `beforeEach` `truncateAll`. Flow over `request(app.getHttpServer())`:
   - `POST /api/auth/magic-link` with a real email → `200 { ok: true }`. To get the raw token (mailed, never returned), read the `loginToken` row via `prisma.loginToken.findFirst`, then reverse it is NOT possible (only the hash is stored) — instead intercept it by generating the token through the seam the service exposes: assert the row exists + `tokenHash` matches `^[0-9a-f]{64}$`, then mint a verify path by stubbing `MailService` in the test module to capture the emitted link (`overrideProvider(MailService)`), parse `?token=` out of the captured link.
   - `POST /api/auth/magic-link` with an **unknown** email → still `200 { ok: true }` (anti-enumeration assertion).
   - `POST /api/auth/verify` `{ token }` → `200`, body `{ user: { id, email, displayName } }`, and `set-cookie` includes `httpOnly` access + refresh cookies (`ACCESS_COOKIE`/`REFRESH_COOKIE` from `cookie.util`).
   - `GET /api/auth/me` with the access cookie (via `.set('Cookie', cookies)`) → `200` with the same `{ id, email }`; without the cookie → `401` (`JwtAuthGuard`).
   - `POST /api/auth/refresh` with the refresh cookie → `200 { ok: true }` and **rotated** cookies (new `set-cookie`); then `POST /api/auth/logout` → `200`, clears both cookies.
5. **Poll e2e (`polls.e2e-spec.ts`).** Reuse the auth helper to obtain an authenticated cookie jar, then exercise the lifecycle:
   - `POST /api/polls` (authed) with a `CreatePollDto` (title + one date + ≥1 slot) → `201`, capture the returned poll `id` and `publicToken`.
   - `GET /api/public/polls/:token` (no auth) → `200`; assert the sanitized shape from `PublicService.findByPublicToken`: has `id/title/dates[].slots[]`, and **MUST NOT** contain `userId` or any `participants`/`email` keys (no-leak assertion).
   - `POST /api/public/polls/:token/responses` with a `SubmitResponsesDto` (`displayName`, optional `email`, `answers[]` referencing real `pollSlotId` strings) → `201 { publicToken }`. Re-submit with the **same email** on the same poll → `409` (`@@unique([pollId, email])` → `ConflictException`). Submit an `answers[].pollSlotId` not on this poll → `400`.
   - `GET /api/public/polls/:token/results` → `200`; assert `best` + `slots[]` per-slot tallies present (live computation), and that `best.slotId` matches the highest-score slot.
   - `POST /api/polls/:id/complete` (authed) with a valid `finalSlotId` → `200`/poll status `completed`; a `finalSlotId` from another poll → `400`.
   - BigInt ids arrive as **strings** on the wire (global `BigIntSerializerInterceptor`) — assert `typeof body.id === 'string'`.
6. **Public-service tie-break unit gap-fill (`public.service.spec.ts`).** Mock `prisma.$queryRaw` to return a pre-ordered `rows` array (the SQL `ORDER BY score DESC, available_count DESC, unavailable_count ASC, event_date ASC, start_time ASC, s.id ASC` does the sort; the service trusts `rows[0]`). Add cases proving `getResults` maps `rows[0]` → `best` and coerces `SUM(...)` bigint/string → `Number`; cover the DESIGN §4 worked example (seed matrix: Bob/Charlie/Diana over [Lunch, Dinner, all-day, Morning]) so the expected top slot/score is asserted, and a zero-response (all-zero tally, LEFT JOIN) slot stays present. Empty poll → `best === null`. Reuse the existing `prisma.$queryRaw` mock convention already in this spec.
7. **Auth-service unit gap-fill (`auth.service.spec.ts`).** Add a `describe('parseTtlToMs / hashing / expiry edges')`:
   - Drive `parseTtlToMs` indirectly through `requestMagicLink`/`createSession` by varying `MAGIC_LINK_TTL`/`REFRESH_TOKEN_TTL` in the existing `config.get` mock (`'45s'`→+45000ms, `'1h'`→+3_600_000ms, `'30d'`→+2_592_000_000ms) and asserting `expiresAt - before` lands in the expected band; assert a malformed TTL (`'15x'`) makes `requestMagicLink` reject (the regex `^(\d+)([smhd])$` throws).
   - Hashing: assert the stored `tokenHash`/`refreshTokenHash` matches `^[0-9a-f]{64}$` (already partly covered — extend to assert determinism: hashing the captured plain token via Node `crypto.createHash('sha256')` equals the stored hash, proving plaintext is never stored).
   - Expiry edges in `verify`/`refresh`: `expiresAt = new Date(Date.now() - 1)` → `UnauthorizedException`; `expiresAt = new Date(Date.now() + 60_000)` → succeeds (mirror existing mocks).
8. **Wire jest-e2e config.** Point `jest-e2e.json` at the new `global-setup-e2e.ts`/`global-teardown-e2e.ts`, add `setupFiles`/`setupFilesAfterEach` only if a per-file hook is needed, and set `maxWorkers: 1` so the parallel specs do not race on the single shared schema. Keep the existing `ts-jest` transform and `testRegex`.

## Execution strategy
- **Fan-out unit:** one agent per authored test file — `auth.e2e-spec.ts`, `polls.e2e-spec.ts`, the `public.service.spec.ts` tie-break gap-fill, and the `auth.service.spec.ts` hashing/expiry gap-fill — each a distinct concern.
- **Shape:** pipeline with a sequential barrier first. The shared e2e DB harness/bootstrap (`jest-e2e.json` wiring + `setup-e2e.ts` + `global-setup-e2e.ts` + `global-teardown-e2e.ts` + the replaced `app.e2e-spec.ts`) is a **sequential prerequisite that MUST exist and boot green before the per-flow e2e specs fan out**; the two unit gap-fill files have no dependency on the harness and may run fully in parallel from the start. After the barrier: find (the lifecycle endpoints) → transform (author specs) → verify (run them).
- **Isolation:** none — agents write distinct test files in the shared tree (no worktree needed); `truncateAll` keeps per-flow data from colliding and `maxWorkers: 1` serializes against the one disposable schema.
- **Verify stage:** each authored spec must actually run green — `npm test` for the two `*.spec.ts` gap-fills and `npm run test:e2e` for the harness + flow specs (both run inside `backend/`). For the high-assurance flows a verifier re-runs the e2e suite and confirms the three security invariants hold as assertions: no participant emails leak in `GET /api/public/polls/:token`, `200`-on-unknown-email for `POST /api/auth/magic-link`, and `409`-on-duplicate-email for `POST /api/public/polls/:token/responses`.
