# Phase 3: Deploy wiring and docs

**Plan:** [resend-email-migration](00-overview.md)
**Depends on:** [02-backend-smtp-config.md](02-backend-smtp-config.md)
**Execution:** solo

## Context
Pollendar's outbound email (magic-link sign-in, poll-completed notices) goes through the existing nodemailer `MailService`, configured entirely by the validated `SMTP_*` / `MAIL_FROM` env vars. The feature goal is to send PRODUCTION email via Resend SMTP from `Pollendar <pollendar@heymanuel.ch>` while keeping Mailpit for local dev and the frontend Playwright e2e suite — a purely env-driven prod-vs-dev split, no code or API changes. Phase 2 already confirmed `MailService`/`env.validation.ts` accept Resend's `secure`/auth-required SMTP shape; this final phase wires the Resend secrets into the production deploy path, updates every doc that mentions SMTP/Mailpit to document the split, and verifies one real magic-link send from `pollendar@heymanuel.ch`.

## Objective
Document and wire the Resend SMTP production env (secrets injected at container runtime), update all SMTP/Mailpit-referencing docs to reflect the prod=Resend / dev=Mailpit split, and verify a real end-to-end magic-link delivery from `pollendar@heymanuel.ch` via Resend.

## Steps
1. **`docs/DEPLOY.md` — rewrite the SMTP rows in the "Required production environment" table.** Replace the existing five rows (`SMTP_HOST`, `SMTP_PORT`, `SMTP_SECURE`, `SMTP_USER`, `SMTP_PASSWORD`) and `MAIL_FROM` with Resend-specific values:
   - `SMTP_HOST` | yes | `smtp.resend.com`
   - `SMTP_PORT` | yes | `465` (with `SMTP_SECURE=true`) or `587` (STARTTLS, with `SMTP_SECURE=false`)
   - `SMTP_SECURE` | yes | `true` for port 465; `false` for port 587
   - `SMTP_USER` | yes | the literal string `resend` (Resend's fixed SMTP username)
   - `SMTP_PASSWORD` | yes | the Resend API key (`re_...`) — inject via secret manager, never bake into the image
   - `MAIL_FROM` | yes | `"Pollendar <pollendar@heymanuel.ch>"` — must be on a Resend-verified domain
   Remove the "optional*"/"required by most real relays" caveat on `SMTP_USER`/`SMTP_PASSWORD`; both are now REQUIRED for Resend. Add a one-line note that dev uses Mailpit instead (`SMTP_HOST=localhost`, `SMTP_PORT=1025`, `SMTP_SECURE=false`, empty user/password) and that switching is purely env-driven — no code change.
2. **`docs/DEPLOY.md` — add a "Resend setup (one-time)" section** (after the "Secrets" section). Write it as a copy-pasteable runbook the user performs in the Resend dashboard + DNS for `heymanuel.ch`:
   1. In the Resend dashboard, **Domains → Add Domain → `heymanuel.ch`**.
   2. Resend shows DNS records to add at the `heymanuel.ch` DNS provider. Add each exactly as shown (record names/values are generated by Resend — copy them verbatim):
      - **MX** record for the bounce/return-path subdomain (e.g. `send.heymanuel.ch`) → value `feedback-smtp.<region>.amazonses.com`, priority `10`.
      - **TXT (SPF)** on that same subdomain → value `"v=spf1 include:amazonses.com ~all"`.
      - **TXT (DKIM)** record(s) → name `resend._domainkey` (or as Resend specifies) → the long DKIM public-key value Resend provides.
      - Optionally the **TXT (DMARC)** record Resend recommends on `_dmarc.heymanuel.ch`.
   3. Back in Resend, click **Verify** and wait until the domain status reads **Verified** (DNS propagation can take minutes to hours).
   4. **API Keys → Create API Key** (sending permission, scoped to the verified domain). Copy the `re_...` key once — it is the value for `SMTP_PASSWORD`.
   5. Confirm the sender identity `pollendar@heymanuel.ch` falls under the verified `heymanuel.ch` domain (Resend verifies the whole domain, so any `@heymanuel.ch` From works).
3. **`docs/DEPLOY.md` — add a "Verify a real send" runbook** (after the Resend setup section) describing the manual end-to-end check used in this phase's Verification (real magic-link from `pollendar@heymanuel.ch` via Resend to a real inbox).
