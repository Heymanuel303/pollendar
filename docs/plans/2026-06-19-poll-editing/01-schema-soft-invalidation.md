# Phase 1: Schema soft-invalidation column

**Plan:** [poll-editing](00-overview.md)
**Depends on:** none
**Execution:** solo

## Context
The poll-editing feature lets a creator edit a live poll WITHOUT destroying participant votes: add dates/slots, soft-invalidate (deactivate) existing dates/slots while preserving their historical votes, edit scalar fields, and cancel/reopen the poll. The locked design uses SOFT invalidation — a date/slot that already has votes is never deleted; instead a nullable `invalidatedAt` timestamp marks it inactive (hidden from voting, excluded from results/tally, rejected on submission) and can be set back to `null` to reactivate. This phase lays the pure data-model foundation: add that column to both `PollDate` and `PollSlot` via an additive (expand-only) migration. No consuming code (services, queries, frontend) changes here — later phases read/write the column.

## Objective
Add a nullable `invalidatedAt DateTime?` column (mapped to `invalidated_at`) to the `PollDate` and `PollSlot` Prisma models and generate the additive Prisma 7 migration that creates the two columns, leaving all existing rows/data untouched.

## Files to touch
- `backend/prisma/schema.prisma` — add `invalidatedAt DateTime? @map("invalidated_at")` to the `PollDate` model (currently lines 107–119) and the `PollSlot` model (currently lines 121–138).
- `backend/prisma/migrations/<TIMESTAMP>_schema_soft_invalidation/migration.sql` — NEW, AUTO-GENERATED by `prisma migrate dev` (do not hand-write). Must contain only two additive `ALTER TABLE ... ADD COLUMN ... DATETIME(3) NULL` statements.
- `backend/prisma/seed.ts` — DO NOT EDIT (see Steps; the new column is nullable with no default, so the existing seed is forward-compatible). Listed only to record that it was checked and intentionally left alone.
- `backend/src/prisma/seed.spec.ts` — DO NOT EDIT (it asserts row counts / graph shape only, never column shape; the additive column does not change any assertion). Listed only to record it was checked.

## Steps
1. Add the new field to the `PollDate` model in `backend/prisma/schema.prisma`. Place it directly after the `createdAt` scalar (line 112) and before the relation fields (`poll`, `slots`), mirroring the project's field ordering (scalars first, then relations, then `@@` blocks). The exact line to add (match the existing 2-space-aligned style of the model):
   ```prisma
   invalidatedAt DateTime?  @map("invalidated_at")
   ```
   Resulting `PollDate` model field block should read, in order: `id`, `pollId`, `eventDate`, `sortOrder`, `createdAt`, `invalidatedAt`, then `poll` relation, `slots` relation, then the existing `@@unique([pollId, eventDate])`, `@@index([pollId])`, `@@map("poll_dates")`. Leave the `@@` constraint/index/map blocks unchanged and in their current order.
2. Add the same field to the `PollSlot` model in `backend/prisma/schema.prisma`. Place it directly after the `createdAt` scalar (line 129) and before the blank line that precedes the relation fields (`date`, `responses`, `tally`, `finalOf`):
   ```prisma
   invalidatedAt DateTime?  @map("invalidated_at")
   ```
   Do NOT add an index on `invalidated_at` — later queries filter by `invalidatedAt: null` together with the parent FK that is already indexed (`@@index([pollDateId])` / `@@index([pollId])`), so a standalone index is unnecessary for this schema size. Leave `@@index([pollDateId])` and `@@map("poll_slots")` unchanged.
3. Generate the migration from the `backend/` directory (where `schema.prisma` and `prisma.config.ts` live):
   ```bash
   cd backend && npx prisma migrate dev --name schema_soft_invalidation
   ```
   This auto-creates `backend/prisma/migrations/<TIMESTAMP>_schema_soft_invalidation/migration.sql`, applies it to the dev `pollendar` DB, and re-runs the generator. Notes for a cold session:
   - `migrate dev` needs the shadow-DB grant (`GRANT ALL PRIVILEGES ON \`prisma_migrate_shadow_db_%\`.* TO 'pollendar'@'%'`) — already applied to the running container (memory: prisma7-setup). If it is missing after a `docker compose down -v`, re-apply it as root/root before running.
   - `migrate dev` is NOT gated by the agent-consent env var — only `migrate reset` is. This migration is purely additive (two nullable columns), so it will NOT prompt for data loss and will NOT reset the DB. Do NOT run `migrate reset` in this phase.
   - The connection string comes from repo-root `.env` via `prisma.config.ts` — no extra env wiring needed.
4. Inspect the generated `migration.sql` and confirm it contains ONLY two additive statements (and nothing else — no table drops, no `MODIFY`, no `DROP COLUMN`). Expected content (column order may differ; both must be present):
   ```sql
   -- AlterTable
   ALTER TABLE `poll_dates` ADD COLUMN `invalidated_at` DATETIME(3) NULL;

   -- AlterTable
   ALTER TABLE `poll_slots` ADD COLUMN `invalidated_at` DATETIME(3) NULL;
   ```
   If Prisma emits anything destructive, STOP — the schema edit was wrong; fix the schema and re-run. Never hand-edit the generated `migration.sql`.
5. Confirm `backend/prisma/migrations/migration_lock.toml` is unchanged (`provider = "mysql"`) — it is auto-managed and read-only; do not touch it.
6. Do NOT change `backend/prisma/seed.ts` or `backend/src/prisma/seed.spec.ts`. Confirmed by reading them: the seed only creates rows (it never sets `invalidatedAt`, which now defaults to `NULL`), and the spec asserts only row counts / null-email / all-day-null-times / the 3×4 response matrix — never column existence or type. A nullable column with no default keeps the seed idempotent (the spec runs `seed()` twice) and keeps every existing assertion green. No consuming code (`polls`/`public` modules, frontend) is touched in this phase — that is Phases 2+.
